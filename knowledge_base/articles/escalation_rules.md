# Правила эскалации обращений

## Автоматическая эскалация

### Критерии эскалации на 2-ю линию

- Проблема не решена в течение 2 часов (для P2)
- Требуется доступ к серверам или базам данных
- Необходима настройка сетевого оборудования
- Проблема требует анализа логов
- Требуется изменение конфигурации системы

### Критерии эскалации на 3-ю линию

- Критический инцидент (P1)
- Проблема не решена в течение 4 часов (для P2)
- Требуется вмешательство разработчиков
- Проблема с архитектурой системы
- Необходима работа с производителем ПО
- Уникальная проблема без известного решения

## Ручная эскалация

Специалист 1-й линии может вручную эскалировать обращение если:
- Проблема явно выходит за рамки компетенции
- Требуются специальные знания или доступы
- Пользователь запрашивает эскалацию

## Атрибуты обращения

При создании обращения автоматически заполняются:

**Обязательные поля:**
- Номер обращения (автоматически)
- Дата и время создания
- Пользователь (из Telegram)
- Тематика (определяется классификатором)
- Критичность (определяется классификатором)
- Линия поддержки (определяется автоматически)
- Статус (Новое)

**Дополнительные поля:**
- Описание проблемы (из сообщения пользователя)
- Контекст диалога
- Предложенное решение (если есть)
- История эскалаций
- Связанные обращения

## Шаблоны обращений

### Шаблон для 1-й линии
```
Тип: Запрос пользователя
Линия: 1-я линия
Приоритет: Средний/Низкий
Категория: [Определяется автоматически]
```

### Шаблон для 2-й линии
```
Тип: Технический инцидент
Линия: 2-я линия
Приоритет: Высокий/Средний
Категория: Техническая проблема
Требуется: Анализ логов, настройка системы
```

### Шаблон для 3-й линии
```
Тип: Критический инцидент
Линия: 3-я линия
Приоритет: Критический/Высокий
Категория: Системная проблема
Требуется: Экспертное вмешательство
```
